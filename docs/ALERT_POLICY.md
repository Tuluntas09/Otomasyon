# ALERT_POLICY.md

## Purpose

This document defines the exact language rules for all alerts generated by Otomasyon.
Alerts are the primary user-facing output of Tier 2 capability. They must describe
threshold crossings in factual terms and must never prescribe action.

---

## Allowed alert language

Alerts may state:
- Which named rule fired.
- Which threshold was crossed and in which direction.
- The current measured value.
- The date / time of the measurement.

**Template:**
> `[RULE NAME] threshold crossed. [Metric] is [value], which exceeds the [direction] limit of [threshold] set in rule [rule-id].`

**Examples (allowed):**
- "Concentration alert: AAPL weight is 32.4%, above the single-position ceiling of 25%."
- "Drawdown alert: portfolio value is 18.2% below its 90-day peak."
- "Volatility alert: 30-day rolling standard deviation of daily returns is 2.8%, above the 2.0% threshold."

---

## Forbidden alert language

Alerts must **never** contain:

| Forbidden word / phrase | Reason |
|---|---|
| buy | Advisory |
| sell | Advisory |
| hold | Advisory |
| reduce | Advisory |
| increase | Advisory |
| recommend | Advisory |
| suggest | Advisory |
| opportunity | Advisory / profit claim |
| profit | Profit claim |
| loss | Predictive / advisory (use "drawdown" or "decline in value" instead) |
| should | Advisory |
| must | Advisory |
| target price | Advisory |
| price prediction | Advisory |
| guaranteed | Prohibited always |

**Examples (forbidden):**
- ❌ "AAPL is overweight — consider selling some."
- ❌ "Portfolio risk is high. You should reduce exposure."
- ❌ "There is an opportunity to profit from rebalancing."
- ❌ "Buy more bonds to reduce volatility."

---

## Compliance guard

All alert strings must pass through the compliance guard module before delivery to the user.
The guard scans for forbidden terms and raises a `ComplianceViolationError` if any are
found. This is a hard gate — bypassing it is not permitted.

*Implementation: Phase 5.*

---

## Rule naming

Each alert rule must have a stable rule ID (e.g., `CONC-001`, `DD-001`, `VOL-001`).
The rule ID must appear in every alert message that fires from that rule, so the user can
trace the alert back to its configuration.

*Rule configuration schema: Phase 5.*
